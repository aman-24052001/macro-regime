1. PROBLEM STATEMENT (Finance Context)
================================================================================

Static allocation strategies (e.g., 60/40 equity-bond) implicitly assume market 
regimes are stable. In reality, macroeconomic conditions shift between expansion, 
stagnation, and contraction — each with fundamentally different risk/return 
profiles across asset classes.

Key failure modes of static strategies:
- 2008 GFC: Equities and credit collapsed simultaneously; bonds rallied but 
  60/40 still drew down ~30%
- 2020 COVID crash: V-shaped recovery punished those who de-risked too late
- 2022 rate shock: Bonds and equities fell together, breaking the traditional 
  diversification assumption

Core question: Can we systematically detect macroeconomic regime shifts in 
near-real-time and dynamically rotate factor/asset exposure to improve 
risk-adjusted returns and reduce tail risk?

2.3 Data Alignment & Preprocessing Pipeline
--------------------------------------------------------------------------------

Problem: Mixed-frequency data — GDP is quarterly, CPI monthly, yield curve daily.

Solution:
1. Resample all features to monthly frequency (last business day)
2. Quarterly features (GDP): forward-fill then apply cubic spline interpolation 
   OR use monthly GDP proxy (e.g., INDPRO as real activity proxy)
3. Daily features (spreads): take end-of-month or monthly average
4. Apply vintage-aware alignment — use FRED ALFRED API to pull data as it was 
   known at time T (eliminates look-ahead bias from GDP revisions)
5. Handle missing data: forward-fill with max 3-month gap, drop features with 
   >10% missing in training window
6. Standardize all features to rolling 36-month Z-scores (prevents distribution 
   shift across decades)

3. METHODOLOGY (Statistical / Algorithmic Detail)
================================================================================

3.1 Hidden Markov Model — Theory & Design
--------------------------------------------------------------------------------

Why HMM over alternatives?
- HMMs capture latent states that generate observed macro data — directly 
  analogous to unobservable "economic regimes"
- Unlike threshold rules (e.g., "recession if yield curve inverts"), HMMs learn 
  regime boundaries from data
- Unlike simple clustering (K-means), HMMs model temporal persistence — regimes 
  are sticky, not random

Model Specification:
- Type: Gaussian Hidden Markov Model (GaussianHMM from hmmlearn)
- Number of hidden states: 3 (determined via BIC/AIC comparison across 2-5 states)
  - State 1 — Expansion: High growth, low spreads, steep yield curve, low unemployment
  - State 2 — Stagnation/Transition: Mixed signals, flattening curve, rising uncertainty
  - State 3 — Contraction: Inverted/flat curve, widening spreads, rising unemployment, falling production

Model Parameters (learned via EM algorithm):
- pi (initial state distribution): Prior probability of starting in each regime
- A (transition matrix): 3x3 matrix of regime-to-regime transition probabilities 
  — captures persistence and shift likelihood
- mu_k, Sigma_k (emission parameters): Mean vector and covariance matrix of 
  macro features for each regime k

Covariance type: "full" (allows feature correlations within each regime — 
critical because macro features are correlated)

Training:
- EM algorithm (Baum-Welch) with multiple random initializations 
  (n_iter=200, n_fits=50) to avoid local optima
- Select best fit by log-likelihood


3.2 Model Selection & Validation
--------------------------------------------------------------------------------

Number of states selection:
- Train HMMs with K = 2, 3, 4, 5 states
- Compare using BIC (Bayesian Information Criterion) — penalizes complexity
- Cross-validate against NBER recession dates as ground truth (regime 3 should 
  align with NBER contractions)
- 3 states is expected winner — 2 loses transition nuance, 4+ overfits

Walk-Forward Validation Protocol:
1. Initial training window: 1985-2005 (20 years)
2. First out-of-sample month: Jan 2006
3. Expanding window: Each month, re-train HMM on all data up to T-1, predict 
   regime at T
4. No future data ever leaks into training — this is critical for credibility
5. Continue through 2024

Regime Detection Latency Analysis:
- For each NBER recession start date, measure how many months after the official 
  start the HMM first assigns P(Contraction) > 0.5
- Target: detect regime shift within 1-3 months (acceptable), >4 months (model 
  is too slow)
- Compare latency against simple heuristic (yield curve inversion signal)


3.3 Regime Probability Extraction
--------------------------------------------------------------------------------

Instead of hard regime assignment (argmax), extract smoothed posterior 
probabilities for each regime at each time step using the forward-backward 
algorithm.

Output at each month T:
- P(Expansion | data up to T)
- P(Stagnation | data up to T)
- P(Contraction | data up to T)

These probabilities become the allocation signal.

4. PORTFOLIO CONSTRUCTION & ALLOCATION LOGIC
================================================================================

4.1 Probability-Weighted Dynamic Allocation
--------------------------------------------------------------------------------

NOT binary switching. Use regime probabilities to blend across regime-specific 
portfolios.

Regime-Specific Target Portfolios:

Asset               | Expansion | Stagnation | Contraction
--------------------|-----------|------------|------------
Equities (SPY)      | 60%       | 30%        | 10%
Long Treasuries(TLT)| 10%       | 25%        | 40%
Gold (GLD)          | 5%        | 15%        | 25%
Corp Bonds IG (LQD) | 15%       | 15%        | 5%
High Yield (HYG)    | 10%       | 5%         | 0%
Cash (BIL)          | 0%        | 10%        | 20%

Final allocation at month T:

  w_T = P(Expansion_T) x w_expansion 
      + P(Stagnation_T) x w_stagnation 
      + P(Contraction_T) x w_contraction

This produces smooth, gradual allocation shifts — no violent portfolio turnover.


4.2 Rebalancing Rules
--------------------------------------------------------------------------------

- Frequency: Monthly (end of month, after regime probabilities update)
- Threshold rebalancing: Only trade if any asset weight deviates >2% from target 
  (reduces unnecessary turnover)
- Transaction cost model: 5bps per trade (one-way) — conservative for liquid ETFs
- Slippage: Additional 2bps assumed
- No leverage, no shorting (keeps it realistic for institutional long-only mandates)


4.3 Position Sizing Refinement — Volatility Scaling (Optional Enhancement)
--------------------------------------------------------------------------------

Within each asset's weight, apply inverse-volatility scaling using 21-day 
rolling realized volatility:

  w_i_adjusted = w_i_target x (sigma_target / sigma_i_realized)

Capped at 1.5x target weight to prevent concentration. This ensures each 
position contributes roughly equal risk.

8. KNOWN LIMITATIONS (Show Self-Awareness)
================================================================================

State these explicitly in the README and be ready to discuss in interviews:

1. Look-ahead bias risk — even with ALFRED, some indicators are revised months 
   later. GDP vintage data partially mitigates but doesn't eliminate.

2. Regime count assumption — 3 states is a modeling choice. Real economies have 
   more nuanced states (stagflation, reflation, goldilocks). BIC selection is 
   a reasonable but imperfect heuristic.

3. Stationarity assumption — HMM assumes transition probabilities are constant 
   over time. In reality, structural breaks (e.g., post-2008 QE era) may shift 
   regime dynamics.

4. Survivorship bias in ETFs — ETF data only starts mid-2000s. Pre-ETF 
   backtests use index proxies which ignore real trading frictions.

5. No forward-looking alpha claim — this is a risk management / drawdown 
   reduction tool, not an alpha generator. Frame it as such.

6. US-centric — model trained on US macro data; applying to global portfolios 
   requires additional validation.

7. Regime labels are post-hoc — HMM states are numbered, not named. Labeling 
   them "Expansion/Contraction" is interpretive, based on examining emission 
   parameters.
